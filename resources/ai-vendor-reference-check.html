A2X field kit · vendor diligence

Check AI vendor references for real workflow evidence, not applause.

Use this worksheet after a promising demo and before a pilot commitment. It helps a buyer, champion, or workshop attendee ask references about day-to-day usefulness, adoption friction, support, safety boundaries, and exit paths without collecting private details.

Reference-safety rule

Ask for patterns, not secrets.

Do not request customer names, private metrics, contract terms, pricing details, screenshots, confidential workflows, regulated records, employee data, production prompts, private repository links, or incident specifics. Ask for generic before/after patterns, roles, review gates, support behavior, adoption friction, and what they would verify again.

Step 1

Prepare the reference call

1. Workflow to validate

Name the one job the vendor claims to improve: intake triage, meeting prep, ticket summary, document review, code support, knowledge search, stakeholder updates, or another repeated workflow.

2. Proof you need

Decide what would make the reference meaningful: fewer handoffs, clearer review packets, faster first drafts, better retrieval, lower support friction, safer escalation, or easier rollback.

3. Boundaries to test

List what your first pilot would not touch yet: live customer data, production systems, payments, credentials, HR records, regulated data, external sends, or irreversible actions.

4. Follow-up decision

Choose the possible outcome before the call: run a tiny trial, clarify one risk, ask for another reference, or park the vendor for now.

Step 2

Ask reference questions that reveal operating fit

Workflow reality

“What repeated workflow did you actually use it for, and what artifact or decision improved? Please keep examples generic.”

Adoption friction

“What was harder than expected during setup, training, permissions, review habits, or getting people to use the workflow consistently?”

Human review and safety

“Where does a person review, approve, pause, or override the output? What inputs or actions did you keep off-limits at first?”

Support and ownership

“Who owns configuration, support questions, incident review, access changes, and rollback decisions now that the tool is in use?”

Evidence quality

“What evidence convinced you to continue, narrow, pause, or expand the workflow? What evidence would you collect earlier if starting again?”

Exit path

“If you had to stop using it, what would be easy or hard to export, archive, replace, explain to users, or clean up?”

Step 3

Translate the call into a decision

Run a tiny trial

Reference matched your workflow

They described a similar job, realistic adoption friction, explicit review gates, named ownership, useful evidence, and a clean exit path.

Clarify first

Reference was useful but incomplete

Ask one more bounded question, request a second reference, or tighten the pilot boundaries before spending team time.

Park it

Reference depended on risky assumptions

Pause if the value required private data, broad permissions, unclear ownership, weak review gates, hidden support burden, or no credible rollback path.

Advanced artifact workflow

Presentation editor overview.

A safe workshop pattern for turning outlines into editable slide artifacts with an agent. Use it to teach the workflow without exposing private demos, internal prototypes, or raw source repositories.

Public-safe framing

Demo the loop, not the private implementation

The useful attendee lesson is the agentic editing loop: provide a slide goal, generate a draft, inspect the artifact, give targeted feedback, and repeat. Keep direct source links and private document-to-slides prototypes out of the public hub until their setup and safety posture are deliberately curated.

Step 1

Start from a slide contract

Define audience, outcome, tone, number of slides, source notes allowed, and non-goals before asking an agent to produce anything visual.

Step 2

Generate a reviewable draft

Ask for a structured outline or HTML slide draft first. Prefer something easy to inspect in a browser before moving into polished deck tooling.

Step 3

Edit with precise feedback

Review slide-by-slide. Request concrete changes: tighten the title, split a dense slide, add a transition, or replace vague claims with sourced wording.

Fictional mini-brief for the workshop

Goal: create a 6-slide intro deck for a fictional AI operations workshop.
Audience: product and engineering leads who have tried coding agents once.
Tone: practical, optimistic, no hype.
Required slides:
1. Why harnesses matter
2. Rules and repo context
3. Tool access and verification
4. Human review gates
5. Reusable skills
6. Next exercise
Non-goals: pricing claims, customer logos, live private notes, or production deployment promises.

Agent prompt for the basic pattern

Create a first-pass slide outline from this brief.
Return Markdown with one section per slide.
For each slide include:
- slide title
- speaker intent
- 3-5 bullets
- suggested visual metaphor
- review questions for the facilitator

Use only the fictional brief above. Do not add customer names, metrics, pricing, or unsupported claims.

Workshop-safe setup guidance

  • If a live generator uses a model API, say clearly whether attendees need their own key or whether the facilitator is demoing from a prepared environment.
  • Use placeholders such as [GEMINI_API_KEY] or [MODEL_API_KEY]; never include real keys in slides, docs, chat, or screenshots.
  • Do not upload private workshop notes, customer decks, internal strategy, unpublished metrics, or local files during a public demo.
  • Prefer browser-inspectable artifacts for teaching. Once the feedback loop is clear, the same pattern can be adapted to PowerPoint, Google Slides, or custom editors.
  • Keep source-repo links approval-gated until README/setup copy, example data, and secrets scans are clean enough for attendees.

90-second facilitator script

  1. Show the fictional mini-brief and name the constraints before generation.
  2. Generate a plain outline, not a perfect deck.
  3. Review the draft by slide purpose: “what is this slide trying to make the audience do?”
  4. Give one targeted revision request, then show how the artifact improves.
  5. Explain that production deck workflows need the same gates as code: source control, secret hygiene, review, and rollback.

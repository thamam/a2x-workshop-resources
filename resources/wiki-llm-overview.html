Knowledge workflow

Wiki-LLM overview.

A safe pattern for turning a small set of Markdown notes into a model-assisted wiki. Use fictional notes in the workshop; keep real vaults private.

Core idea

Separate private knowledge from reusable workflow

The publishable idea is not someone’s actual vault. It is the architecture: keep source notes in Markdown, ask an agent to build linked wiki pages, and preserve an operating contract so future agent runs know how to update the system safely.

Layer 1

raw/ source notes

Short meeting notes, paper summaries, transcripts, or snippets. These stay close to the original material and should be treated as private unless explicitly sanitized.

Layer 2

wiki/ synthesized pages

Agent-written concept pages with backlinks, concise explanations, examples, open questions, and citations back to the raw notes.

Layer 3

Operating contract

A small rules file that tells the agent how to add pages, preserve source links, avoid unsupported claims, and flag uncertain facts instead of inventing them.

Tiny fictional mini-vault

vault/
  raw/
    meeting-prep-assistant.md
    reusable-agent-rules.md
  wiki/
    meeting-prep-assistant.md
    agent-harness.md
  OPERATING_CONTRACT.md

Example raw note:

# Meeting prep assistant
Fictional workshop idea: a consultant wants a tool that turns agenda, attendees, and prior notes into a pre-call brief.
Constraints: no private client data in demos; output should include risks, questions, and next actions.

Agent prompt for the basic pattern

Use the Markdown vault at <vault-path>.
Read OPERATING_CONTRACT.md first.
Convert the notes in raw/ into concise wiki pages under wiki/.
For each page, include:
- summary
- source notes used
- related pages
- open questions
- unsupported claims to verify
Do not invent facts. If a detail is missing, write "unknown" and explain what source would resolve it.

Workshop safety checklist

  • Use fictional or sanitized notes, never live private vault content.
  • Keep setup examples generic: <vault-path>, [TOKEN], and [plugin-path].
  • Prefer Markdown + git as the first source of truth; no vector database or cloud store is required for the basic exercise.
  • Separate software/plugin instructions from vault content so public workflow code does not expose private knowledge.
  • Review generated wiki pages for unsupported claims before sharing them.

When to show this live

  1. After explaining that agents need durable context, not only bigger prompts.
  2. When attendees ask how to reuse research, meeting notes, or project decisions across sessions.
  3. As an advanced example after the simpler CLAUDE.md, skill, and prompt-improvement resources land.
Static public-safe overview. No network calls, tracking, storage, private vault names, private repository links, local paths, or real tokens.

Safe demo rule

Ask for proof on a sanitized version of your real workflow.

Do not send private data, customer files, credentials, internal documents, production repository links, account screenshots, vendor contracts, or sensitive tickets to make a demo feel realistic. Use fictional or redacted samples, then judge whether the tool can show useful behavior within the same boundaries you would allow in a first safe trial.

Step 1

Write the demo brief before the call

1. Workflow we care about

Name one repeated job: triage support themes, prepare a project handoff, review a PRD, draft a stakeholder update, summarize meeting notes, or compare tool options.

2. Safe sample input

Bring a toy example, redacted transcript, synthetic ticket set, copied non-sensitive doc, or fictional mini-project. If the demo requires private input to look good, that is evidence too.

3. Decision evidence

Decide what would count as useful proof: cleaner artifact, fewer missing questions, traceable citations, faster review packet, better handoff, lower support load, or safer approval path.

4. Boundaries

State what the demo must not touch: secrets, live customer data, payments, production systems, private repos, external sends, employee records, or irreversible actions.

Step 2

Ask questions that reveal trial fit

Workflow fit

“Show the tool doing this exact workflow on the sanitized sample. What does the human review, edit, or approve before anything leaves the sandbox?”

Evidence and observability

“What artifact, log, diff, citation, reviewer queue, or before/after comparison will prove whether the tool helped?”

Boundary handling

“What happens if the input contains something off-limits, incomplete, conflicting, or unsafe? Can the tool pause and ask instead of guessing?”

Setup and ownership

“Who owns configuration, permissions, support, incident review, and rollback if this becomes a small pilot?”

Exit path

“If the trial fails, how do we export notes, remove access, archive decisions, and avoid leaving a half-adopted workflow behind?”

Step 3

Decide the next move

Run a tiny trial

Evidence was specific

The demo handled the safe sample, showed review gates, produced a useful artifact, and can run with reversible access and named ownership.

Clarify first

Demo looked promising but vague

Ask for one more bounded proof point, define the use case, or score the trial before scheduling team time.

Park it

Demo depended on unsafe assumptions

Pause if value requires private data, production access, pricing commitments, weak review gates, or unclear accountability before evidence exists.

Copyable demo note

Paste this into your planning doc

Vendor/tool: [name]

Workflow to demonstrate: [one repeated job]

Safe sample: [toy / redacted / fictional input]

Off-limits: secrets, customer data, production systems, private links, external sends, payments, employee records, and irreversible actions.

Proof requested: [artifact, log, review packet, citation, or before/after comparison]

Follow-up decision: tiny trial / clarify first / park.

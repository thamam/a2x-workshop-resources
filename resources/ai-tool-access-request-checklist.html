Access rule

Access is not the first step; scoped use is.

Before adding a user, account, workspace, integration, or paid seat to an AI tool, write the smallest approved use, the data boundary, the human review gate, and who can pause access. Keep this checklist generic: no tokens, account screenshots, customer records, employee data, production logs, private repository links, billing pages, internal architecture, vendor contract terms, or regulated records.

Access request ingredients

Six details every request should include

Use case

What job needs access?

  • Name one repeated workflow or workshop exercise.
  • Describe the output the requester expects to review.
  • Keep it narrower than “general productivity.”
Allowed inputs

What may enter the tool?

  • List approved categories such as toy repos, fictional samples, or redacted notes.
  • Name what is explicitly off limits.
  • Route sensitive-use requests to the data-boundary checklist.
Account route

Which account path is intended?

  • Clarify personal, company, workspace, vendor, or sandbox route.
  • Record who approves the route without sharing secrets.
  • Use credential routing for login confusion.
Review gate

Who reviews outputs?

  • Name the reviewer role before any output is used.
  • Define pass, revise, escalate, and stop decisions.
  • Link access to the output review rubric when needed.
Support owner

Who handles friction?

  • Assign support for setup, policy questions, and broken workflows.
  • Set a fallback if access is delayed or denied.
  • Document where issues should be raised safely.
Offboarding

How does access end?

  • Set a review date, pause trigger, and cleanup owner.
  • Remove users, integrations, files, or workspaces when scope ends.
  • Escalate unclear retention or deletion questions.
How to use it

Run a ten-minute access review before the first login

Start with the use case

Ask what workflow, workshop exercise, or pilot task needs the tool. If the answer is broad, narrow it to one reversible first use with a clear reviewer.

Check the input boundary

Write allowed input categories and the do-not-paste list. If real customer, employee, regulated, production, contract, or repository material is required, stop and route to the appropriate approval path.

Choose the account path

Decide whether this is a sandbox, company workspace, vendor-provided account, internal assistant, or personal-learning route. Do not paste credentials, tokens, billing pages, or screenshots into the request.

Attach support and exit

Name who helps when access fails, who reviews output, when access is revisited, and what trigger pauses or removes it. Access without cleanup is just future archaeology.

Good next moves

Where access governance fits

Before setup

Use the credential routing Q&A to troubleshoot login, provider, token, workspace, and policy confusion without exposing secrets or billing/account screenshots.

Before real data

Use the pilot data boundary checklist and policy exception request when the access request depends on sensitive inputs, integrations, retention, or legal/security review.

Before routine use

Pair access with the workflow owner card, output review rubric, and vendor exit plan so accounts do not outlive purpose, ownership, or safe-use boundaries.

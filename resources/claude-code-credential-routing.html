Safe rule

Never paste tokens, account pages, billing screens, or customer data into a shared prompt.

Credential routing is a local setup question, not a workshop demo asset. Ask “which route is this machine using?” before asking “why did the model fail?” and keep the evidence to non-secret status messages.

Fast diagnosis

Four routes to distinguish

1 · Browser login

Claude Code is tied to an interactive account session.

Look for login prompts, expired browser sessions, workspace selection, or a machine that has never completed the local sign-in flow.

2 · API key

A token or provider key is configured locally.

Confirm the key exists only on the attendee machine, is not screen-shared, and is not copied into files, chats, screenshots, or workshop repos.

3 · Environment override

An environment variable may be taking precedence.

Shell profiles, IDE terminals, and wrapper scripts can point a session at a different account or provider than the attendee expects.

4 · Organization policy

The account may be valid but blocked by policy.

Managed devices, enterprise settings, network rules, and billing/workspace permissions can prevent use even when login appears successful.

Triage flow

Use this order before debugging prompts

Reproduce in a non-sensitive folder. Open Claude Code in a toy repo or empty scratch folder so diagnostics do not touch private files.

Read the visible status, not the secret. Capture only the tool’s non-secret error text: logged out, unauthorized, quota, model unavailable, network blocked, or permission denied.

Name the intended route. Decide whether this setup should use browser login, API key, managed workspace, or a facilitator-owned demo machine.

Check overrides locally. Inspect terminal profile, IDE launch environment, and wrapper scripts on the attendee machine without exposing token values to the room.

Escalate or switch modes. If account policy blocks progress, use the live demo and continue with the harness map, project rules, product brief, and PRD exercises.

Copyable Q&A prompts

Questions Tomer can ask without collecting secrets

What route did you expect?

“Were you expecting this machine to use browser login, a workspace account, or a locally configured API key?”

What safe evidence do we have?

“What non-secret error or status does Claude Code show? Please do not show token values, account pages, or billing screens.”

Can we isolate the environment?

“Does the same issue happen from a clean terminal in a scratch folder, outside the IDE or project wrapper?”

What is the fallback?

“If this is a policy or billing issue, which planning resource can you still use today without local Claude Code access?”

Do not do this live

Keep secrets and account ownership out of the workshop artifact.

  • Do not commit .env files, shell history, screenshots of account pages, or token-shaped strings.
  • Do not ask attendees to paste credentials into prompts, shared docs, chat, or issue comments.
  • Do not troubleshoot billing, enterprise policy, or production access in front of the room; switch to the facilitator demo or non-code exercises.
  • Do not treat a credential failure as a prompt-quality failure. Fix the route first, then evaluate the workflow.

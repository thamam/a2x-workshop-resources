A2X field kit · change readiness

Check AI tool changes before workflows drift.

Use this when a model, vendor feature, tool policy, account setting, integration, or internal rule changes after an AI workflow is already in use. The goal is not a heavy audit. It is a fast public-safe check: what changed, who is affected, what proof must be rerun, and whether the workflow should continue, narrow, retrain, pause, or escalate.

Change-impact rule

Describe the change with sanitized categories, not private evidence.

Do not paste customer data, employee records, contracts, regulated data, production logs, account screenshots, private repository contents, credentials, vendor pricing, raw prompts, incident transcripts, or internal strategy. Record the change type, affected workflow category, owner, safe test sample, review result, and decision boundary.

Step 1

Name what changed

Model or behavior

Output changed

The model version, tool behavior, retrieval pattern, formatting, latency, refusal behavior, or confidence level changed enough to rerun review samples.

Access or policy

Boundary changed

Allowed inputs, accounts, retention, sharing, vendor terms, internal policy, or reviewer requirements changed before routine use continued.

Integration

Workflow changed

A connector, automation step, prompt template, permissions group, or handoff path changed and may affect what users see or approve.

Incident or drift

Trust changed

A near miss, support spike, quality drop, user workaround, or unclear ownership signal suggests the workflow needs a pause-and-check moment.

Step 2

Run the lightweight impact check

Change summary

Change type: [model/tool/policy/integration/access/support/incident]. Observed by: [role/forum]. Date noticed: [date]. Affected workflow: [generic workflow category].

Users and boundary

Included users: [role/team category]. Allowed inputs: [sanitized input categories]. Human review gate: [reviewer/forum]. Boundary question: [what might need approval].

Proof to rerun

Safe sample: [toy/redacted/fictional sample]. Expected output: [reviewable artifact]. Reviewer checks: [quality/safety/compliance/support]. Pass signal: [specific visible evidence].

Support and communication

User notice needed: [none/update/retrain/pause note]. Owner: [role]. Support route: [office hours/channel/forum]. Documentation update: [prompt/rule/checklist/handoff].

Decision

Recommended next state: continue, update, retrain, narrow, pause, stop, or escalate. Decision owner: [role/forum]. Revisit trigger: [new change, failed review, incident, support load, policy response].

Step 3

Choose the safest next state

Continue

The safe sample still passes review, the data boundary is unchanged, users do not need new training, and the owner can monitor the next cycle.

Update or retrain

The workflow still works, but prompts, checklist copy, enablement notes, reviewer instructions, or support guidance need a refresh before routine use.

Narrow or escalate

The change creates an unclear policy, access, vendor, support, or data-boundary question. Keep the workflow smaller while a named reviewer decides.

Pause or stop

The safe sample fails, boundary drift appears, ownership is unclear, users are confused, or a near miss suggests routine use should halt until reviewed.

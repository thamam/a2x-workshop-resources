A2X field kit · pilot judgment

Decide what “good enough” means before the AI workflow runs.

Use this handout after the use-case intake, data-boundary checklist, experiment charter, or assumption log. It helps a team define acceptable outputs, reviewer actions, evidence thresholds, and stop rules before pilot results get judged by vibes.

Acceptance rule

If a reviewer cannot tell pass from revise, the pilot is not ready to score.

Write acceptance criteria as observable behavior, not hope. Keep examples generic, fictional, redacted, or approved. Do not paste customer data, employee records, production logs, private screenshots, credentials, regulated data, contracts, pricing, raw prompts, private repository links, incident details, or confidential strategy.

Step 1

Name the output and the decision it supports

Artifact

What the AI produces

Define the draft, summary, checklist, code change, analysis, recommendation, or routing decision the workflow is expected to create.

User action

What a human does next

Name the action the output should support: approve, edit, send, compare, escalate, test, archive, or ask for more evidence.

Failure consequence

What can go wrong

Identify what an unacceptable output could break: trust, safety, policy, customer experience, reviewer time, or the next pilot decision.

Step 2

Define four judgment states

Pass

The output is accurate enough, complete enough, safe enough, and clear enough for the named human next step after normal review.

Revise

The output is useful but needs a small correction, missing field, tone fix, source check, formatting pass, or extra reviewer note.

Escalate

The output raises an unclear policy, data, legal, security, customer, procurement, or operational boundary that needs approval before use.

Pause or stop

The output crosses a boundary, repeats unsafe errors, cannot be reviewed reliably, or asks for sensitive data the pilot is not approved to handle.

Step 3

Use AI to draft criteria without leaking context

Decision prompt: “Using only the public-safe workflow description below, draft acceptance criteria for an AI-assisted workflow. Return pass, revise, escalate, and pause/stop criteria; reviewer evidence to collect; and common failure modes. Do not invent private examples, customer names, employee data, production logs, pricing, approvals, credentials, or private repository details.”

Human next step: [review / revise / approve / test / send / escalate / decide].

Pass criteria: [observable signs the output is accurate, complete, safe, and usable after review].

Revise criteria: [small issues that should be corrected before use].

Escalate criteria: [approval, policy, data, legal, security, customer, or operations boundary].

Pause/stop criteria: [unsafe content, repeated failure, unreviewable output, sensitive-data request, or boundary drift].

Do not include: customer records, employee data, production logs, account screenshots, credentials, regulated data, vendor contracts, pricing claims, private repository links, raw prompts, incident transcripts, or internal strategy.

When to use this

Good moments for the handout

Before the first pilot run

Agree on pass/revise/escalate/stop states before the first output is emotionally persuasive.

Before evidence logging

Make reviewer notes comparable so the evidence log can show patterns instead of anecdotes.

Before rollout readiness

Check whether the workflow has stable criteria before a small trial expands to more users.

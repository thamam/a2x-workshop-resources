A2X field kit · pilot safety

Draw the data boundary before an AI pilot touches real work.

Use this checklist before a trial, pilot, or internal workflow experiment starts using team data. It helps attendees define allowed inputs, redaction rules, reviewer access, retention, cleanup, and escalation without pasting private examples into shared docs.

Boundary rule

If the data category is unclear, it is out of scope.

Do not use customer records, employee data, credentials, regulated records, production logs, private repository contents, vendor contracts, account screenshots, or confidential strategy in a pilot until the owner, reviewer, allowed transformation, retention window, and escalation path are explicit and approved.

Step 1

Name the data the pilot may touch

1. Allowed input

Write the safest acceptable input class: fictional examples, redacted support tickets, copied non-sensitive docs, toy repository files, public docs, or synthetic meeting notes.

2. Off-limits input

List what is not allowed yet: secrets, tokens, private repo URLs, raw customer data, HR records, financial records, regulated records, contracts, account pages, or production logs.

3. Redaction standard

Define exactly what must be removed or generalized before use: names, emails, IDs, domains, dates, screenshots, ticket links, file paths, and business-sensitive numbers.

4. Owner and reviewer

Name who approves a sample before it enters the pilot and who can pause the pilot if the boundary is unclear or violated.

Step 2

Set operating controls before the first run

Access

Who can see inputs and outputs?

Keep the pilot group narrow. Note whether outputs may be copied into tickets, docs, Slack, email, demos, or only the private pilot workspace.

Retention

How long do examples stay around?

Pick a cleanup date for sample files, exported outputs, screenshots, prompts, and review notes. Archive only sanitized evidence.

Escalation

What happens when the boundary fails?

Stop the run, preserve non-sensitive context, notify the owner, remove exposed content from shared docs, and update the checklist before retrying.

Step 3

Use the boundary matrix

Green: safe to use now

Fictional, synthetic, public, or fully redacted examples where no person, customer, account, secret, internal system, or contract can be reconstructed.

Yellow: clarify before use

Partially redacted samples, internal process docs, low-sensitivity tickets, or copied snippets that need owner review before they enter the model/tool workflow.

Red: keep out of the pilot

Secrets, access tokens, production incidents, legal/HR/finance records, regulated data, customer-identifying material, private source links, vendor contracts, and screenshots with account details.

Exit rule

An exit plan is not pessimism; it is trial hygiene.

Before a team depends on an AI tool, write the plain-language exit path: what data must be exported or deleted, how users keep working, who decides to stop, and what evidence proves the transition is safe. Keep this worksheet generic: no contract terms, account screenshots, vendor secrets, customer records, internal architecture diagrams, pricing, or regulated data.

Exit-path ingredients

Six questions to answer before commitment

Workflow dependency

What would break if the tool vanished?

  • Name the workflow, user role, and human fallback.
  • List downstream decisions that rely on its output.
  • Mark what can pause safely versus what needs continuity.
Data return

What needs to come back?

  • List export categories, not sensitive records.
  • Define owner, format, and timing expectations.
  • Separate audit evidence from operational content.
Data cleanup

What must be removed or retained?

  • Name retention and deletion questions for review.
  • Flag logs, prompts, files, users, and integrations.
  • Route policy/legal/security details to approved owners.
Replacement path

How will work continue?

  • Pick manual, existing-tool, or narrowed-scope fallback.
  • Define which users need a transition note.
  • Write the first safe workarounds before the trial scales.
Decision owner

Who can call exit?

  • Name roles for stop, export, cleanup, and communication.
  • Set the evidence threshold for stop versus renew.
  • Include a backup owner so silence is not approval.
Trigger signals

When do we leave or renegotiate?

  • Mark safety, quality, support, access, reliability, or policy triggers.
  • Include renewal-review timing before urgency arrives.
  • Keep pricing claims and private contract details out of the public worksheet.
How to use it

Run a thirty-minute exit-path review

Start from one workflow

Do not plan a generic vendor divorce. Pick the workflow being tested, the user group, the allowed sample boundary, and the decision this tool is supposed to support.

Map the minimum continuity path

Write how the work would continue for one week without the tool: manual fallback, existing system, pause-and-communicate, or narrowed scope.

Write export and cleanup questions

Use categories such as prompts, outputs, files, users, configuration, audit logs, and integration state. Do not paste real records, screenshots, contract terms, credentials, or private system paths.

Attach exit to a decision meeting

Bring the plan into the demo brief, reference check, decision meeting, or renewal review so exit readiness is discussed before enthusiasm, deadlines, or sunk cost take the wheel.

Good next moves

Where exit planning fits

Before the demo

Use the demo brief to ask how the vendor supports export, cleanup, fallback, audit evidence, and user transition without discussing confidential contract details.

Before the decision

Bring unresolved exit questions into the decision meeting agenda, pilot risk register, or policy exception request instead of burying them in procurement folklore.

Before renewal

Compare exit readiness with evidence logs, value scorecards, support load, and tool-change impact before deciding to renew, narrow, replace, or stop.

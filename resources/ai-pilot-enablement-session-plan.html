Enablement rule

A good session teaches the workflow, the guardrail, and the stop button.

Do not train users with customer records, employee data, contracts, regulated data, production logs, account screenshots, private repository contents, credentials, vendor pricing, incident transcripts, or internal strategy. Use a toy task, redacted pattern, or fictional scenario that proves the workflow shape without importing sensitive material.

Step 1

Decide whether this session is ready to run

Workflow

One repeatable task

The pilot has a named workflow, intended users, allowed inputs, expected output, and a human review gate.

Evidence

Enough proof to teach

There is sanitized evidence from a small trial, demo, or dry run that shows what works, what fails, and what remains off-limits.

Support

Owners are named

The session can name the workflow owner, reviewer, support route, escalation path, pause authority, and next review date.

Boundary

No hidden sensitive examples

The teaching sample can be fictional, redacted, or synthetic. If real sensitive material is required, stop and route to policy or security review first.

Step 2

Run a 45-minute enablement agenda

0–5 min · Why this workflow exists

Name the repeated pain, the pilot goal, the included users, and the decision boundary. Avoid broad AI capability claims.

5–12 min · Show the safe example

Walk through a fictional or sanitized input, expected output, and review step. Say clearly what data is not allowed.

12–22 min · Practice one run

Let users try the toy task or watch a facilitator-run demo. Emphasize the prompt boundary, review gate, and evidence to capture.

22–30 min · Explain human ownership

Name who approves output, who handles support, who updates rules, and who can pause or narrow the pilot.

30–38 min · Handle questions and failure modes

Collect confusion, support friction, risky inputs, and likely misuse as categories rather than private examples.

38–45 min · Confirm next behavior

State the allowed next action, feedback channel, evidence format, next review date, and pause trigger before people leave.

Step 3

Choose the session outcome

Launch to tiny group

Users understood the workflow, data boundary, review gate, support path, and pause rule. Start only with the named group and next review date.

Run one more practice

The workflow is promising, but users need a clearer example, better prompt, stronger support path, or narrower allowed-input rule.

Pause rollout

The session revealed unclear ownership, risky data needs, weak review gates, or unsupported users. Pause and return to readiness, policy exception, or incident-pause material.

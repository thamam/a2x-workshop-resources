Safe evidence rule

Log categories and outcomes, not confidential material.

Keep the evidence log shareable: record workflow names, dates, counts, pass/fail notes, redacted examples, reviewer decisions, and open questions. Do not paste customer data, employee records, account screenshots, production credentials, private repository links, vendor contracts, raw logs, or unredacted incident transcripts.

What counts as evidence

Capture proof points that can change a decision

Value

Useful work happened

Time saved, fewer handoff errors, faster review, better draft quality, or clearer next actions — recorded with enough context to be inspected.

Friction

Adoption had a cost

Setup difficulty, confusing prompts, review burden, policy questions, support asks, or workflow steps that still required manual rescue.

Risk

Boundaries were tested

Near misses, sensitive-input pressure, hallucinated confidence, stale instructions, unsafe outputs, or cases where the pilot should stop and clarify.

Checklist

Seven fields for a useful pilot log

1. Name the workflow and trial window

Use a generic workflow name, owner, users involved, start date, and review date. Avoid client names or private repository identifiers.

2. State the hypothesis

Write what improvement the pilot is supposed to prove, such as faster first draft, cleaner handoff, fewer review cycles, or better triage.

3. Record each observed run

For every run, note date, safe input type, output type, pass/fail result, reviewer, and one sentence of sanitized context.

4. Separate value from risk

Keep wins, friction, quality issues, safety concerns, and support questions in separate rows so one dramatic example does not distort the decision.

5. Capture reviewer decisions

Record what a human reviewer accepted, edited, rejected, escalated, or blocked — including why.

6. Mark open questions

Name the policy, ownership, data, support, or technical question that must be answered before rollout, operation, or retirement.

7. Point to the next artifact

Decide whether the log should feed a trial scorecard, decision memo, risk register, stakeholder update, retrospective, rollout checklist, health review, or sunset packet.

Facilitator cue

When to use it in the workshop

After scorecard

Turn a promising trial into proof

Use the log immediately after the scorecard so attendees know what evidence to collect before asking for a bigger commitment.

Before decision memo

Avoid vague recommendations

Have the team fill the evidence log before writing “continue” or “stop” so the memo has observed facts, not preference theater.

During health review

Keep recurring reviews grounded

Use the same log format for operational pulse checks so value, risk, friction, and support signals stay comparable over time.

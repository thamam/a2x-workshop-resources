How to use OpenSpec with agents.

OpenSpec is a lightweight way to turn “please build this” into a reviewable change proposal before implementation starts. It helps the agent ask better questions, make smaller changes, and preserve why decisions were made.

Plain English

Spec before code, but not bureaucracy

The useful habit is simple: before an agent edits code, ask it to write the change as a proposal with motivation, scope, acceptance criteria, and impact. Then approve, revise, or split it.

Step 1

Write the intent

Describe the user problem, desired behavior, non-goals, constraints, and known risks.

Step 2

Draft a proposal

Ask the agent for a concise proposal before code: why, what changes, impact, tests, and open questions.

Step 3

Review the proposal

Check whether the change is too broad, missing acceptance criteria, or relying on invented facts.

Step 4

Implement against it

Once the proposal is good, tell the agent to implement only that scope and verify the acceptance criteria.

Minimal proposal template

# Change proposal: [short name]

## Why
What user or business problem does this solve?

## What changes
- [concrete change]
- [concrete change]

## Non-goals
- [what not to build now]

## Impact
Files, users, workflows, data, integrations, or risks affected.

## Acceptance criteria
- Given..., when..., then...
- Test or manual verification step...

## Open questions
Only questions that materially affect implementation.

Agent prompt

You are helping me prepare an OpenSpec-style proposal before implementation.
Use the brief below.
Do not write code yet.
Return:
1. Why
2. Proposed changes
3. Non-goals
4. Impact
5. Acceptance criteria
6. Risks
7. High-impact missing questions
8. Recommendation: proceed, split, or clarify first

[paste brief]

Decision gates

  • Proceed: goal, scope, acceptance criteria, and risks are clear enough.
  • Clarify: one or two missing decisions could change the implementation.
  • Split: the proposal mixes unrelated changes or is too large to verify.
  • Stop: the work touches private data, production systems, credentials, money, or irreversible actions without explicit approval.

Why it helps workshops

OpenSpec gives attendees a visible checkpoint between idea and implementation. They can inspect the agent’s assumptions before the codebase changes, then use acceptance criteria as the verification plan.

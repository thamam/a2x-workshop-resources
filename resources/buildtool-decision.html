The Decision

Extend the agent's core tool surface?

Your A2X skills (especially resilient-flows and code-research) are already sophisticated orchestrators. The question is whether to go one level deeper and contribute first-class native tools using the buildTool factory from the Claude Code source.

The specific proposal
Add SpecReviewFinding as a native tool in resilient-flows
The review agents in spec-review-remediate currently emit findings as free text. A real tool would make findings a first-class, typed, observable capability.
High leverage

With a first-class buildTool

The model gains SpecReviewFinding as a real tool in its native toolbox — exactly like Bash or FileEdit.

New flow
1. Model sees tool
Schema + description appear in prompt via .prompt()
2. Agent calls tool
Structured args during review (file, severity, specRef, evidence)
3. Engine handles
UI rendering • hooks fire • transcript indexed • permissions applied • subagent context passed
The skill changes too:
  • Review agents instructed to call the tool instead of describe findings
  • Orchestrator consumes structured tool results + any side effects written to state
  • Findings become first-class data the entire resilient-flows system can react to

Tradeoffs & Cost

More code to own
You now maintain a tool definition, renderers, and registration logic instead of pure prompt orchestration.
Registration surface
Need to expose the tool through the plugin system or dynamic loading (see how other plugins contribute tools in the source).
High one-time learning, high ongoing leverage
The Tool interface is large (~100 methods/fields), but most of the power is in a handful of advanced methods.

Implementation Roadmap

1. Define the tool
Create resilient-flows/tools/SpecReviewFindingTool.ts using the full buildTool contract. Start minimal and add advanced methods (extractSearchText, preparePermissionMatcher, renderers) iteratively.
2. Update the skill
In spec-review-remediate, change agent instructions from "describe findings" to "use the SpecReviewFinding tool". Update scoring and remediation phases to consume tool results.
3. Wire registration
Make the tool appear when the resilient-flows plugin is active (plugin contribution points or a small loader that augments the tool registry).
4. Enhance your hooks (optional but powerful)
Your existing precompact_snapshot.py and Bash guards can now react specifically to this tool. Add a PostToolUse handler for findings if desired.

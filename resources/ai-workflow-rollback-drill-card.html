Rollback rule

If the team has not rehearsed the fallback path, the rollback path is still a wish.

Keep this page public-safe. Use toy, fictional, synthetic, redacted, or approved examples only. Do not paste customer data, employee records, production logs, credentials, account screenshots, regulated data, vendor contracts, pricing, raw prompts, incident transcripts, private repository links, or confidential strategy.

Step 1

Name the smallest rollback scenario

Trigger

What makes us stop?

Choose one observable trigger: unsafe output, boundary confusion, review backlog, access change, support spike, stale model behavior, or owner absence.

Fallback

What do users do instead?

Define the manual, previous-tool, or narrower workflow path people can use while the AI-assisted workflow is paused.

Restart

What proves it can resume?

Name the safe sample, reviewer signal, cleanup evidence, and decision owner required before restarting or expanding the workflow.

Step 2

Run a 20-minute tabletop drill

Call the pause

Say who notices the trigger, who can pause, which users are affected, and what channel carries the first message.

Switch to fallback

Walk through the manual or narrower path using a toy example, then note any missing access, instruction, owner, or support detail.

Collect recovery evidence

Record the safe sample rerun, reviewer result, cleanup action, user-support note, and remaining risk without exposing sensitive artifacts.

Decide next state

Choose continue, narrow, retrain, update rules, escalate, pause longer, or retire. Assign one owner and one review date.

Step 3

Use AI to draft a safe drill plan

Rollback-drill prompt: “Using only the public-safe workflow summary below, draft a 20-minute rollback drill for this AI-assisted workflow. Include pause trigger, decision owner, user message, fallback path, recovery evidence, cleanup step, restart condition, and next-state options. Use generic roles and toy examples only. Do not invent customer names, employee data, production logs, credentials, private repository links, vendor contracts, pricing, incident details, or internal strategy.”

When to use this

Good moments for the drill

Before controlled rollout

Use it after rollout readiness so rollback is practiced before more users rely on the workflow.

After a tool or policy change

Use it when a model, vendor, access, or policy change might make the old guardrails stale.

During health review

Use it when support friction, incidents, reviewer drift, or stale evidence suggests the workflow needs a recovery rehearsal.

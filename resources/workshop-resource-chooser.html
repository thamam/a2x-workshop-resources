Suggested path

A simple attendee sequence

Prepare a safe sandbox. Use the pre-workshop checklist if you are not sure what project, sample input, or verification signal to bring.

Route credential confusion. Use the credential routing Q&A if login, account, API-key, environment, or policy issues appear before the first run.

Open the harness map. Anchor on why the surrounding workflow matters as much as the model.

Add project rules. Use the CLAUDE.md cheat sheet to make the next agent run more deliberate.

Package repetition. Build a first skill, then run Skill Wizard as a quick quality gate.

Shape an idea. Generate a product brief, ask only high-impact missing questions, then draft a PRD/OpenSpec starter.

Choose one follow-up workflow. Use the post-workshop action plan to commit to a small safe project with verification evidence.

Package the next run. Use the handoff packet to carry outcome, safe inputs, rules, stop conditions, and verification into the next Claude Code session.

Prioritize candidate workflows. Use the prioritization matrix when several AI ideas compete for attention and you need the safest useful next trial.

Preflight the prompt. Use the prompt safety preflight before pasting context into Claude, Claude Code, ChatGPT, internal assistants, or vendor demos.

Review the AI output. Use the output review rubric before relying on AI-generated drafts, summaries, recommendations, code changes, or workflow artifacts.

Calibrate reviewers. Use the output calibration session when several people need shared pass thresholds, stop conditions, and evidence rules for AI-generated work.

Capture meeting actions. Use the meeting-to-actions handout after workshops, pilot reviews, vendor demos, or steering meetings so AI-assisted summaries become reviewed, reversible next steps.

Escalate blocked boundaries. Use the policy exception request when a useful workflow needs tool, data, legal, security, procurement, or operating-policy approval before it can proceed.

Define the AI use case. Use the intake brief to name the workflow, safe sample input, review gate, evidence signal, and stop condition.

Set the data boundary. Use the data boundary checklist to decide allowed inputs, redaction rules, reviewer access, retention, cleanup, and escalation before a pilot touches workflow data.

Charter the pilot experiment. Use the experiment charter to define the smallest reversible test, safe input, review gate, evidence question, and stop condition before collecting pilot proof.

Log pilot assumptions. Use the assumption log to name dependencies, evidence checks, owner roles, revisit triggers, and fail responses before hidden beliefs become operating rules.

Set workflow acceptance criteria. Define pass, revise, escalate, and pause/stop criteria before reviewer evidence is collected or a pilot result is scored.

Check workflow handoff gaps. Before routine use, find missing ownership, review, support, monitoring, pause, rollback, and next-review details.

Drill the rollback. Practice who pauses the workflow, what users do instead, what recovery evidence is needed, and who decides when it can restart.

Assign workflow ownership. Use the owner card to name the accountable owner, backup, reviewer, support route, review cadence, escalation boundary, and pause condition.

Score the next trial. Use the AI tool trial scorecard before committing team attention to a tool or workflow experiment.

Brief demos around proof. Use the AI vendor demo brief before a vendor or internal tool walkthrough so the conversation stays tied to a safe sample, review gates, and evidence.

Check references safely. Use the AI vendor reference check after a promising demo to ask about workflow reality, adoption friction, ownership, review gates, and exit paths without collecting private details.

Log pilot evidence. Capture value, friction, risk, reviewer decisions, and open questions so the next decision is grounded in observed proof.

Score pilot value. Use the value scorecard to decide whether evidence supports continuing, clarifying, narrowing, expanding, pausing, stopping, or escalating.

Write the pilot decision memo. Convert the trial evidence into a continue, clarify, or stop recommendation with boundaries and one reversible next step.

Run the decision meeting. Use the agenda to align stakeholders on continue, clarify, expand narrowly, pause, stop, or archive without exposing sensitive examples.

Register pilot risks. Before scaling the pilot, name the boundary, mitigation, owner, review signal, and stop condition for the obvious risks.

Send the stakeholder update. Summarize status, evidence, top risk, ask, and next review date without forwarding sensitive examples.

Resolve stakeholder concerns. Turn objections into evidence requests, safe boundaries, named owners, and the smallest reversible next step.

Brief adoption change. Before an AI workflow spreads, explain the human change, support path, review gate, pause trigger, and next review date.

Answer adoption questions. Use the FAQ to explain what the workflow does, what stays out, who reviews it, how support works, and when to pause.

Align sponsor decision. When the next step needs sponsor attention, turn pilot evidence into one decision, commitment boundary, owner, and reversible next step.

Set steering cadence. Before decisions scatter, choose the recurring forum, evidence packet, stop rules, owners, and artifacts that keep the AI workflow governed.

Log steering decisions. After the cadence or review, record the decision state, owner, evidence basis, boundary, revisit trigger, and stakeholder note before the meeting disperses.

Run the pilot retrospective. Convert pilot evidence into lessons, guardrail updates, stop/continue decisions, and the next safe experiment.

Check rollout readiness. Before expanding the pilot, confirm evidence, users, allowed inputs, review gates, rollback path, and stakeholder messaging.

Plan pilot enablement. Teach the small rollout group the safe example, guardrails, support path, pause trigger, and next review date before routine use.

Run adoption office hours. After enablement, support users with sanitized questions, toy examples, guardrail reminders, improvement signals, and escalation routing.

Close the feedback loop. Convert sanitized adoption feedback into one reviewed improvement decision before proof snapshots or steering updates.

Summarize adoption proof. After office hours or health review, capture sanitized use counts, review outcomes, support themes, guardrail health, and the next decision.

Check tool changes. When a model, vendor, policy, access, or integration change appears, rerun a safe sample and decide whether to continue, retrain, narrow, pause, or escalate.

Prepare operations handoff. Before routine use, name the owner, support route, monitoring signal, incident trigger, rollback owner, and improvement cadence.

Review pilot health. After handoff, run a recurring pulse check on usage, quality, incidents, support friction, guardrail drift, value, and improvement decisions.

Pause incidents cleanly. Use the incident pause card when the workflow drifts, crosses an unclear boundary, lacks restart authority, or spreads before the guardrails are ready.

Close or archive deliberately. If health review evidence says stop, pause, replace, or retire the pilot, use the sunset checklist to clean up access, notify users, and preserve reusable lessons.

Review before scaling. Improve prompts and review PRDs before moving to optional advanced demos.

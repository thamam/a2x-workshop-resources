Safe intake rule

Define the use case without exposing the sensitive work.

Describe roles, workflow steps, inputs, outputs, and review gates generically. Do not paste customer data, employee records, production credentials, private repository links, account screenshots, vendor contracts, raw tickets, or internal strategy into a shared brief.

Step 1

Name the real workflow

Trigger

When does the work start?

Example: “A product lead has rough meeting notes and needs a draft PRD before engineering review.”

Current path

How is it done today?

Name the handoff, tool, review loop, or manual checklist that already exists. Avoid vague “use AI for productivity” framing.

Pain

What makes it worth testing?

Look for repeated delay, missing context, inconsistent review quality, rework, or decisions made without enough evidence.

Not in scope

What must not change yet?

List approvals, external messages, production systems, sensitive datasets, payment actions, or irreversible changes that are off-limits.

Step 2

Fill the intake brief

Keep each answer short. The point is to make the next trial scorable and safe, not to write a strategy deck.

1. Use-case sentence

We want to help [role] turn [safe input] into [reviewable output] so that [decision or handoff improves].

2. Safe sample input

Use a toy repo, fictional customer, sanitized notes, copied non-sensitive docs, or a tiny synthetic dataset. If the example cannot be sanitized, the use case is not ready for an open workshop trial.

3. Human review gate

Name who checks the output, what they check, and what would make them reject or revise it before anyone relies on it.

4. Success evidence

Pick one observable signal: cleaner diff, fewer missing questions, faster review packet, accepted outline, repeatable checklist, lower rework, or clearer stop/continue decision.

5. Stop condition

Write the condition that ends the trial: needs sensitive data, cannot be reviewed, output is unverifiable, setup blocks users, risk rises, or the value signal is too weak.

Step 3

Triage the next move

Ready to score

The workflow is specific, the sample is sanitized, the output is reviewable, and the stop condition is clear. Move to the AI tool trial scorecard.

Clarify first

The user, output, or evidence is vague. Use the product brief generator, OpenSpec interviewer, or post-workshop action plan before scoring a tool.

Park for approval

The use case needs private data, production permissions, external sending, procurement, compliance review, pricing decisions, or executive approval before safe evaluation.

Use case: Help [role] turn [safe input] into [reviewable output] so that [decision/handoff improves].

Current workflow pain: [delay / rework / missing context / inconsistent quality / unclear decision].

Safe sample: [toy repo / sanitized notes / fictional customer / copied non-sensitive docs].

Human review gate: [reviewer + acceptance/rejection check].

Success evidence: [artifact, before/after comparison, review result, or decision signal].

Off-limits: secrets, customer data, employee records, production systems, private links, external sends, payments, and irreversible actions.

Next move: score now / clarify first / park for approval.

Guided tutorial

A2X Marketplace tutorial.

A practical, attendee-friendly path for understanding the marketplace idea: install only curated items, load plugins deliberately, and choose skills or plugins based on the job.

Public-safe version

Use this before any direct source link

This tutorial explains the workflow without exposing private setup paths, unreviewed plugin internals, real credentials, or live repository links. Treat it as the attendee-facing install mental model until the marketplace release path is curated.

Quick install mental model

  1. Start from a curated marketplace item. Read its purpose, scope, permissions, inputs, and verification steps before installing anything.
  2. Copy or install into your local agent workspace. Use the item’s documented target such as <project>/.claude/, <home>/.claude/, or <plugin-dir>. Avoid global installs unless the item says why.
  3. Restart or reload the agent harness. Most agents discover new plugins, commands, or skills at startup. If a reload command exists, use that; otherwise start a fresh session.
  4. Run the smallest safe smoke test. Ask the agent to list the loaded item or run its dry-run command before giving it private files or write access.
  5. Verify output and permissions. Keep the item only if it saves repeat work and its side effects are clear.
Item type

Skills

Use when the value is a repeatable procedure: code review, planning, debugging, release checks, document creation, or team-specific habits. Benefit: the agent remembers the right workflow without a long prompt every time.

Item type

Plugins

Use when the agent needs new capabilities: commands, integrations, file transformations, browser actions, or domain tools. Benefit: the workflow becomes executable, not just advisory.

Item type

Presets

Use when a whole environment should feel consistent: model choice, rules, allowed tools, starter skills, and safety defaults. Benefit: teams can start from a known operating mode.

Item type

Templates

Use when the output shape matters: PRD sections, review packets, meeting briefs, acceptance criteria, or launch checklists. Benefit: results become easier to compare and review.

Item type

Commands

Use when the same action should be triggered in one phrase, such as generating a plan, exporting a feedback packet, or running a local audit. Benefit: less prompt drift.

Item type

Safety checks

Use before publishing, sharing, committing, or sending data to tools. Benefit: the agent has explicit gates for secrets, private paths, external links, and destructive actions.

Plugin loading checklist

  • Read the plugin’s permissions and expected files.
  • Check whether it calls a network service, needs an API key, or writes files.
  • Install into a test project first, not an important production repo.
  • Open a fresh agent session and ask what plugin commands/tools are visible.
  • Run the documented dry run using fictional or sanitized data.
  • Only then use real project context, and still review every proposed change.

Safe first exercise

Goal: Try a marketplace skill without risk.
1. Pick a skill that only reads text and returns advice.
2. Paste a fictional product brief.
3. Ask the agent to use the skill and produce a review checklist.
4. Confirm the output cites only the pasted fictional brief.
5. Decide whether the skill saved enough repeated prompting to keep.

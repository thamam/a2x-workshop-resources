Procurement rule

Answer with categories, controls, owners, and evidence — not secrets.

Prepare what the reviewer needs to decide: workflow purpose, allowed input categories, review gates, retention expectations, admin ownership, support route, and open gaps. Keep this worksheet generic: no customer records, employee data, production logs, credentials, account screenshots, private repository links, vendor contract terms, pricing, regulated records, incident transcripts, private architecture, or confidential strategy.

Questionnaire lanes

Sort questions before answering them

Use case
What business workflow is being evaluated?

Name the repeated workflow, user role, expected output, human review point, and decision this evaluation supports.

Data boundary
What input categories are allowed?

List approved categories, do-not-use categories, redaction rules, and where unclear data gets escalated.

Access and admins
Who can use or administer the tool?

Identify generic roles, access request path, account owner, reviewer, offboarding trigger, and support route.

Review and evidence
How will output be checked?

Define pass/revise/stop criteria, reviewer role, required logs or screenshots, and evidence that stays safe to share.

Retention and exit
What happens if the tool is paused or rejected?

Name export categories, cleanup owner, fallback workflow, communication path, and stop or renegotiate triggers.

Open gaps
Which answers need approval?

Route unknown legal, security, data, procurement, or policy questions to named reviewers instead of guessing.

Safe answer pattern

Use this structure for each questionnaire item

1 · Translate

Restate the question in plain language

  • What decision is the reviewer trying to make?
  • Which workflow, data, user, or control is in scope?
  • Which details are unsafe to paste into the shared draft?
2 · Answer by category

Use approved patterns, not raw details

  • Allowed inputs: generic categories.
  • Controls: review gates and owner roles.
  • Evidence: safe artifacts or links to approved internal records.
3 · Mark confidence

Separate known, assumed, and blocked

  • Known: verified by policy, owner, or artifact.
  • Assumed: needs confirmation before submission.
  • Blocked: requires reviewer input or vendor answer.
4 · Escalate gaps

Do not invent a control

  • Send open questions to security, legal, procurement, sponsor, or tool owner.
  • Keep confidential evidence in approved channels.
  • Pause the evaluation if the gap changes the risk boundary.

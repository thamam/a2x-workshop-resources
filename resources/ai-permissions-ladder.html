Permissions rule

Autonomy should rise one rung at a time.

Start with observe or draft-only permissions. Add higher rungs only after the team can name the safe input boundary, reviewer, approval moment, rollback path, and pause authority. Keep this worksheet generic: no customer records, employee data, production logs, credentials, private repository links, account screenshots, vendor contract terms, regulated records, incident transcripts, pricing, or confidential strategy.

Five permission rungs

Choose the highest safe level for this workflow

Rung 0 · Observe
AI can read approved safe material and summarize what it sees.

Use for discovery, meeting notes, public docs, toy repos, or sanitized examples. Human decides every next action.

Rung 1 · Draft
AI can produce a draft, checklist, analysis, or plan for review.

Human review is mandatory before anyone treats the output as advice, a task, a message, or a code change.

Rung 2 · Edit in sandbox
AI can make changes only in a disposable, reversible, non-production workspace.

Use for workshop repos, local branches, copied docs, or fixture data. Require visible diff, test, or screenshot evidence before accepting.

Rung 3 · Propose external action
AI can prepare a message, ticket, configuration, or integration change, but cannot send or apply it.

A human approves the exact action, audience, destination, and rollback path. This is a proposal rung, not an execution rung.

Rung 4 · Execute with approval
AI can execute a bounded action only after explicit human approval and verified guardrails.

Use rarely. Require named approver, scoped command/action, allowed environment, live stop trigger, rollback owner, and post-action evidence.

Approval gates

Do not climb when any gate is missing

Input boundary

What can the AI touch?

  • List allowed materials by category.
  • Name the do-not-use list.
  • Route unclear data to the data-boundary checklist.
Reviewer

Who decides pass or revise?

  • Name the reviewer role.
  • Define evidence required before acceptance.
  • Keep unresolved items out of execution.
Action scope

What exact action is allowed?

  • Specify draft, edit, proposal, or execution.
  • Limit destination, audience, files, or systems.
  • Ban broad “do whatever is needed” authority.
Escalation path

Where does uncertainty go?

  • Name the policy, security, legal, sponsor, or owner route.
  • Keep sensitive details out of the shared request.
  • Pause instead of improvising around approvals.
Rollback

How do we undo or pause?

  • Define fallback work and cleanup owner.
  • Record what evidence proves recovery.
  • Set a restart decision rule.
Stop trigger

What makes the AI stop?

  • Missing reviewer, sensitive input, failed test, unclear owner, policy concern, or user impact.
  • Stop triggers should be visible before work starts.
  • Higher rungs need stricter stops.

Good next moves

Where permission setting fits

Before ownership handoff

Pair the ladder with the workflow owner card so every permission rung has an accountable owner, backup, reviewer, support route, and pause authority.

Before external action

Use the policy exception request when the next rung would touch sensitive inputs, integrations, communications, procurement, legal, security, or operating-policy boundaries.

Before routine use

Use output review, rollback drills, and health reviews to prove the current rung is stable before granting broader execution authority.

How to work with an LLM Wiki.

An LLM Wiki is a small knowledge base that an agent can read, update, and reason over. The trick is to separate raw notes from synthesized pages, then give the agent an operating contract.

Core pattern

Raw notes in, linked knowledge out

Do not start with a database. Start with Markdown, a tiny folder structure, and a clear rule: the agent may summarize and link, but it must cite sources and flag uncertainty.

Recommended folder structure

vault/
  raw/
    meeting-prep-assistant.md
    reusable-agent-rules.md
  wiki/
    index.md
    meeting-prep-assistant.md
    agent-harness.md
  OPERATING_CONTRACT.md

raw/ holds original notes. wiki/ holds cleaned concept pages. OPERATING_CONTRACT.md tells the agent how to work safely.

Stage 1

Collect notes

Add short raw notes, transcripts, research snippets, or decisions. Keep them private unless deliberately sanitized.

Stage 2

Synthesize pages

Ask the agent to create concise pages with summaries, backlinks, examples, source notes used, and open questions.

Stage 3

Review claims

Read the generated pages. Unsupported claims should be marked unknown, not polished into false certainty.

Stage 4

Maintain over time

When adding new notes, ask the agent what pages should change instead of regenerating the whole wiki blindly.

Starter operating contract

You are maintaining this Markdown wiki.
Rules:
- Read raw/ before editing wiki/.
- Every wiki page must list source notes used.
- Link related pages with Markdown links.
- If a fact is not supported by raw notes, mark it as unknown.
- Do not include private names, credentials, local paths, or raw secrets in public pages.
- Prefer small edits over full rewrites.
- End with a changelog of files changed and claims that need human review.

Prompt to update the wiki

Read OPERATING_CONTRACT.md.
Use the new notes in raw/.
Update wiki/ so it reflects the new information.
For each changed page, include:
1. What changed
2. Source notes used
3. Related pages
4. Open questions
5. Unsupported claims to verify
Do not invent facts. Keep private material private.

When this is useful

  • Research projects with many notes.
  • Meeting-heavy teams that need durable decisions.
  • Product discovery where assumptions change over time.
  • Agent projects that need memory richer than one prompt.

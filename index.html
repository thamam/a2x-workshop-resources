Workshop flow

How to use this hub live

1. Prepare a safe sandbox.

Use the pre-workshop checklist to bring a toy repo, sanitized sample, starter rules, and verification signal.

2. Choose the right next resource.

Open the route chooser if attendees need a fast map from workshop moment to handout.

3. Route setup confusion safely.

If Claude Code login, token, environment, or policy questions appear, use the credential-routing Q&A before debugging prompts.

4. Open with the harness idea.

Show that better agent results come from environment, rules, tools, feedback loops, and review gates.

5. Give attendees rules they can use today.

Send them to the CLAUDE.md cheat sheet before live coding.

6. Turn repetition into infrastructure.

Use the first-skill guide when moving from tips to reusable workflows, then run Skill Wizard to inspect whether Claude can actually discover and use the drafted skill.

7. Hand out practical prompt tools.

Use the product brief generator, interviewer, prompt improver, and PRD starter when attendees start shaping their own ideas; fall back to the synthetic meeting-prep scenario if they need a safe demo input.

8. Close with one next workflow.

Send attendees to the post-workshop action plan so they leave with a small reversible Claude Code project, not just a pile of links.

9. Package the next agent run.

Use the handoff packet to capture safe inputs, stop conditions, and evidence before the next Claude Code session starts.

10. Prioritize candidate workflows.

Use the prioritization matrix when attendees have several AI ideas and need to choose the safest useful next trial.

11. Set the pilot data boundary.

Use the data boundary checklist to define allowed inputs, redaction, reviewer access, retention, cleanup, and escalation before touching workflow data.

12. Charter the experiment.

Use the experiment charter to turn the selected use case into one tiny reversible test with safe input, review evidence, and a stop condition.

13. Score the next trial.

Use the AI tool trial scorecard when attendees need to choose which workflow deserves the next small experiment.

14. Brief vendor demos around proof.

Use the demo brief before a vendor or internal tool walkthrough so the conversation stays tied to a safe sample, review gates, and evidence.

15. Check references safely.

Ask references about workflow reality, adoption friction, support ownership, review gates, and exit paths without collecting private details.

16. Log pilot evidence.

Collect value, friction, risk, reviewer decisions, and open questions before asking a team to continue, expand, or stop.

17. Score pilot value.

Use the value scorecard to turn observed evidence into a continue, clarify, narrow, expand, pause, stop, or escalate call.

18. Write the pilot decision memo.

Turn the scored trial into a continue, clarify, or stop recommendation with evidence, boundaries, and one reversible next step.

19. Run the decision meeting.

Use the agenda to choose continue, clarify, expand narrowly, pause, stop, or archive based on evidence, boundaries, ownership, and one reversible next step.

20. Register pilot risks.

Name the boundary, mitigation, owner, review signal, and stop condition before any pilot expands beyond a reversible test.

21. Send a stakeholder update.

Share the pilot status, evidence, top risk, ask, and next review date without forwarding sensitive examples.

22. Answer adoption questions.

Use the adoption FAQ when teams ask what the workflow does, what stays out, who reviews it, how users get help, and when it pauses.

23. Align the sponsor.

Turn pilot evidence into one sponsor decision, commitment boundary, owner, and reversible next step before asking for more attention.

24. Set the steering cadence.

Name the recurring decision forum, evidence packet, stop rules, owners, and artifacts that keep AI workflow decisions from drifting.

25. Run the pilot retrospective.

Turn pilot evidence into lessons, guardrail updates, honest stop/continue decisions, and one next safe experiment.

26. Check rollout readiness.

Confirm evidence, users, allowed inputs, review gates, rollback path, and stakeholder messaging before the pilot expands.

27. Plan pilot enablement.

Teach users and reviewers the safe example, guardrails, support path, pause trigger, and next review date before treating rollout as adoption.

28. Reinforce adoption with office hours.

Run a lightweight support rhythm for sanitized questions, guardrail reminders, improvement signals, and escalation routing after enablement.

29. Close the feedback loop.

Use sanitized adoption feedback to choose one reviewed improvement before proof snapshots, health reviews, or steering updates.

30. Prepare operations handoff.

Name the owner, support route, monitoring signal, incident trigger, rollback owner, and improvement cadence before routine use.

31. Triage the review queue.

Use the review queue triage sheet when AI outputs are arriving faster than humans can safely judge them.

32. Pause incidents cleanly.

Use the incident pause card when an AI workflow drifts, crosses an unclear boundary, lacks an owner, or spreads before guardrails are ready.

33. Park advanced demos safely.

Use placeholders for tools until public review is complete.
